# J the Agent

`J` — модульный AI-агент для Python 3.11+ с единым универсальным ядром.
Режимы — **отдельные продукты**: установщик ставит ровно один из них. Они
зависят от `core`, но **не зависят друг от друга**.

- **Quick** — диагностика и починка Linux/Unix-сервера;
- **Companion** — личный помощник для macOS/Linux с памятью, web-поиском,
  desktop-инструментами и напоминаниями.

Manager будет добавлен позже как ещё один независимый режим.

## Установка

Из клонированного репозитория:

```bash
./scripts/install.sh
```

С GitHub:

```bash
bash <(curl -fsSL https://raw.githubusercontent.com/inafield/J-The-Agent/main/scripts/install.sh)
```

Скрипт предлагает стрелками выбрать Quick или Companion (Manager пока
недоступен), затем:

1. использует уже установленный Python 3.11+ (сам Python не устанавливает);
2. создаёт отдельный venv в `~/.local/share/j-the-agent`;
3. копирует в установку только `core` + выбранный режим (второй режим не ставится);
4. ставит зависимости и команды `ja` / `agent` в `~/.local/bin`;
5. записывает выбранный режим в install manifest;
6. запускает мастер настройки этого режима.

Системные Python-пакеты не изменяются. Чтобы сменить продукт (Quick ↔ Companion),
удалите текущую установку и запустите `install.sh` снова.

**Официальная установка только через `scripts/install.sh`.** Отдельного
`pip install j-the-agent` для пользователей нет: pip внутри скрипта ставит
зависимости в приватный venv после того, как скрипт уже скопировал `core` и
выбранный режим. Выбор Quick/Companion — меню `install.sh`, не extras pip.

## Запуск, активация и завершение

```bash
ja       # запустить интерактивную сессию
agent    # та же команда, длинное имя
```

У установленной версии вручную активировать venv не нужно: `ja` и `agent`
сразу запускают правильный интерпретатор и выбранный при установке режим.
Интерактивная сессия активна, пока открыт CLI-процесс. Чтобы завершить её,
введите `exit`/`quit` или нажмите `Ctrl-C`.

Companion не держит постоянно работающий LLM-процесс. Для напоминаний он
регистрирует небольшие пользовательские задания launchd (macOS) или cron
(Linux); при срабатывании показывает уведомление и пытается открыть новый
терминал с напоминанием.

При разработке из исходников (оба режима в checkout; не замена install.sh):

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
export J_AGENT_MODE=companion   # или quick — обязателен, если оба режима на диске
ja
deactivate
```

## Основные команды

```bash
ja                              # интерактивный чат
agent ask "проверь место на диске"
ja ask найди ошибку в nginx       # кавычки для запроса необязательны
ja ask --file /etc/nginx/nginx.conf "найди ошибку"
ja setup                        # полный мастер
ja permissions                  # права доступа
ja switch model                 # сменить provider/model
ja switch dir                   # сменить рабочую директорию
ja switch directory             # полный вариант той же команды
ja info                         # активные настройки и инструменты
ja doctor                       # диагностика установки (Quick и Companion)
ja check-reminders              # Companion: overdue without LLM (also on login)
ja reminders                    # Companion: list reminders (no LLM)
ja reminders list / done ID
ja hello                        # Companion: introduction / update profile
ja allow-local 3000             # Companion: allow localhost:PORT for fetch_url
ja allow-local --list
ja version
ja uninstall                    # remove install, keep config
ja uninstall --purge            # also remove config and plugins
```

Commands work the same via `agent`. Available subcommands depend on the installed
mode; `ja --help` shows the current list.

## Как J работает

Quick и Companion — отдельные продукты на общем ядре: оба наследуют
`core.agent.BaseAgent` и используют ограниченный ReAct-цикл. В одной установке
лежит только выбранный режим; код другого режима не ставится. Общие CLI-хелперы
живут в `modes/common_cli.py` и `modes/runtime.py`, не в `core`.

Цикл агента:

1. получает запрос и файловый контекст;
2. выбирает минимально необходимый инструмент;
3. выполняет его через `SafetyGuard`;
4. анализирует ограниченный observation;
5. отвечает или делает следующий шаг.

Есть лимиты шагов и вызовов инструментов, блокировка повторяющихся одинаковых
вызовов и финальный ответ без инструментов при исчерпании лимита. J сокращает
старый контекст и отправляет модели только релевантную группу инструментов.
Ответы поддерживаемых провайдеров выводятся потоково. Поэтому цикл не может
выполняться бесконечно и не тратит токены на ненужные tool-схемы.

В мастере можно выбрать, показывать ли ход выполнения. Включённый режим
показывает краткий progress, вызовы инструментов и observations. Выключенный
показывает пользовательский prompt и сразу итоговый ответ. Внутренняя скрытая
цепочка рассуждений модели не раскрывается.

В открытом чате команды `switch model` и `switch directory` применяются сразу:
LLM-клиент, инструменты и SafetyGuard пересоздаются без перезапуска процесса.
Изменение конфига из другого терминала также автоматически подхватывается.

## Companion: память и инструменты

Память Companion хранится отдельно от кода установки:

```text
~/.local/state/j-the-agent/companion/memory/user.md
~/.local/state/j-the-agent/companion/memory/soul.md
~/.local/state/j-the-agent/companion/memory/memory.md
```

`user.md` содержит устойчивые сведения и предпочтения пользователя, `soul.md`
задаёт стиль и правила агента, а `memory.md` — долгосрочные факты и открытые задачи.
После установки Companion и по команде `ja hello` проходит короткое English UI
интервью (имя пользователя, имя агента по умолчанию J, язык, timezone-город и т.д.);
результаты пишутся в профиль и в markdown. Агент сам решает, что сохранить дальше.
Просьба вроде «отвечай теплее» может обновить `soul.md`. Автопереноса всей
переписки в память нет.

Для экономии токенов system prompt получает только `tools/index.md`. Категории
`memory`, `files`, `system`, `web`, `apps`, `code`, `reminders`, `media`
открываются через `load_tool_guide`: в чат попадают короткие tips, а в API —
нативные JSON-схемы функций из registry. Markdown не дублирует аргументы.

При настройке web-поиска доступны DuckDuckGo без ключа (бесплатно, менее
стабильно), Brave и Tavily с API key, SearXNG с URL instance или отключение.

## Companion: напоминания

Напоминания создаются обычным запросом, например «напомни через 2 часа» или
«каждый день в 07:00». Метаданные хранятся в:

```text
~/.local/state/j-the-agent/companion/reminders.json
```

На macOS используется пользовательский launchd, на Linux — пользовательский
crontab. При запуске Companion и при логине (`ja check-reminders`) показываются
просроченные напоминания. При удалении J scheduler-задания отключаются; обычный
uninstall сохраняет память и список напоминаний.

## Подробная история

J записывает запросы к LLM, ответы, tool calls, observations, подтверждения,
ошибки, модель и token usage в пользовательский журнал:

```text
~/.local/state/j-the-agent/history.log
```

Файл имеет права `0600`, красиво разделён по сессиям и предназначен только для
пользователя — J никогда не читает его и не использует как память. Значения,
похожие на API-ключи, пароли и токены, маскируются. При этом содержимое файлов,
явно переданных через `--file`/`@path`, является частью LLM-запроса и поэтому
попадает в историю.

Обычный `ja uninstall` сохраняет историю. Только `ja uninstall --purge`
удаляет журнал вместе с конфигурацией.

## Какие файлы передаются с запросом

Для обычного вопроса `--file` не нужен. Оба варианта равнозначны:

```bash
ja ask найди баг
ja ask "найди баг"
```

Кавычки полезны только для shell-символов или чтобы визуально отделить длинный
запрос. `--file` нужен, когда конкретный файл следует отправить модели сразу,
не дожидаясь вызова `read_file`.

Одноразовый запрос:

```bash
ja ask --file app.py --file config.yaml "почему сервис не запускается?"
```

Интерактивный чат:

```text
you › проверь @app.py и @"config files/nginx.conf"
```

Передача устроена так:

- каждый путь сначала проверяет `SafetyGuard`;
- текстовый файл читается локально и вкладывается в user message как
  `<attachment type="file" path="...">…</attachment>`;
- каталог передаётся как ограниченный список имён, а не как архив;
- бинарные файлы не встраиваются — модели передаётся только их путь/статус;
- максимум 10 вложений, 60 000 символов на файл и 120 000 суммарно;
- forbidden, недоступные или находящиеся вне выбранной области файлы не
  передаются вообще.

Без `--file`/`@path` J получает только текст запроса и при необходимости сам
вызывает `read_file`, `list_directory` или `grep_search`.

## Безопасность

Первый setup предлагает пять профилей:

1. **Recommended** — текущая директория, `/var/log`, `/var/www`, `/opt`;
   `/etc` и `/tmp` только для чтения; личные ключи и критические пути закрыты.
2. **Advanced** — почти весь сервер, кроме чувствительных и критических путей.
3. **Current directory only** — только каталог запуска.
4. **System essentials + my forbidden paths** — обязательный минимум запретов
   плюс пути, вводимые по одному.
5. **Only my forbidden paths** — опасный режим для опытных пользователей.

Поддерживаются glob-правила, например `/home/*/.ssh`. Forbidden всегда имеют
приоритет:

- `Full`: разрешено всё, что не запрещено;
- `Selected`: только выбранные корни, кроме forbidden;
- `Current directory`: только одна рабочая директория, кроме forbidden.

Пути канонизируются через `resolve()`, поэтому symlink из разрешённой директории
не даёт доступа к запрещённой. Чтение, листинг, grep, вложения, запись и рабочая
директория проходят через `SafetyGuard`. Запись, patch, restart сервиса и
опасные shell-команды требуют явного подтверждения. Несколько действий модели
можно разрешить одним выбором через меню со стрелками.

`run_command` проверяет cwd, явные пути, shell-токены и опасные паттерны,
ограничивает timeout и вывод. Read-only команды (`ls`, `cat`, `grep`, `ss`,
`df`, `ps`, `journalctl`, `systemctl status`, `git diff` и другие) выполняются
без лишнего подтверждения. Удаление, перезапись, restart и package operations
всегда требуют решения пользователя. Это практическое усиление, но не полноценная
OS-песочница: абсолютная изоляция произвольного shell требует firejail,
bubblewrap, Docker или отдельного пользователя/контейнера.

## Инструменты Quick

- файлы: `read_file`, `list_directory`, `grep_search`, `write_file`,
  `patch_file`;
- shell/system: `run_command`, `get_system_info`, `disk_usage`,
  `listening_ports`;
- сервисы: `service_status`, `restart_service`, `failed_services`,
  `journal_tail`;
- `current_time`;
- плагины из `~/.config/j-the-agent/plugins`.

В `core/tools.py` находятся только универсальные контракты и реестр. Реальные
инструменты Quick находятся в `modes/quick/tools.py`, а Companion — в
`modes/companion/tools.py`. Память, scheduler, web и tool guides также остаются
внутри `modes/companion/`; универсальное ядро не зависит от конкретного режима.

## Инструменты Companion

- память: чтение и изменение `user.md`, `soul.md`, `memory.md`;
- файлы и код: чтение, поиск, patch, git status/diff;
- system: команды, процессы, диск и сведения об ОС;
- web: поиск, чтение страницы, открытие URL;
- apps/media: запуск приложений, clipboard, screenshot;
- reminders: создать, перечислить, завершить или отменить напоминание;
- плагины из общей пользовательской папки.

## Удаление

```bash
ja uninstall
# либо, если CLI повреждён:
./scripts/install.sh uninstall
```

Удаление читает `install-manifest.json` и удаляет только созданные установщиком
venv, зависимости внутри него, исходники установки, `agent`/`ja` и manifest.
Python, git и ранее существовавшие системные пакеты не удаляются. Конфигурация,
журнал, память и reminder database сохраняются; `--purge` удаляет также
принадлежащие J конфигурацию, плагины, историю и данные Companion. Scheduler-
задания Companion удаляются в обоих случаях, чтобы они не вызывали отсутствующий
CLI после uninstall.

## Конфигурация и плагины

Конфиг: `~/.config/j-the-agent/config.yaml`, права `0600`. Переменные окружения
используют префикс `J_AGENT_`, например:

```bash
J_AGENT_LLM__MODEL=qwen3:8b ja
```

Пример конфига — `config.example.yaml`; пример плагина — `plugins/example.py`.
